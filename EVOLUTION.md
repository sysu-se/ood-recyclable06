# EVOLUTION.md

## Homework 3 设计演进说明（Hint + Explore）

本文总结我在前面作业基础上，对领域模型 `Sudoku / Game` 的演进。

---

## 1. 我如何实现提示功能？

按照作业要求，我将提示能力拆成两类，并都落在领域接口上，而不是写在 UI 临时逻辑里。

### 1.1 候选提示（Candidate Hint）

- 在 `Sudoku` 中新增：
  - `getCandidates(row, col)`：返回指定空格的候选集合（按行/列/宫约束计算）。
- 在 `Game` 中新增：
  - `getCellCandidates(pos)`：会话级封装，探索模式下自动作用于探索局面。
- 在 UI 语义中：
  - 当 Note 关闭时，Hint 走“下一步提示”；
  - 当 Note 开启时，Hint 改为“候选提示”：
    1. 清空当前可编辑格（若原来有用户填入值）；
    2. 读取候选集合；
    3. 以 Note 形式写回该格。

### 1.2 下一步提示（Next-Step Hint）

- 在 `Sudoku` 中新增：
  - `findNextHint()`：先尝试 **Naked Single**，再尝试 **Hidden Single（行/列/宫）**。
- 在 `Game` 中新增：
  - `getNextHint()`
  - `applyNextHint()`
- Hint 次数策略：
  - 仅在提示**成功落子**时扣减；
  - 无推定值时不扣减，并给出用户提示。

### 1.3 与 Note 的一致性处理

- “Hint 结果被旧 Note 覆盖”的问题：
  - 下一步提示成功后会清除目标格 Note，确保数字显示优先。
- “数字与 Note 显示优先级”问题：
  - 只要格子数字 `value !== 0`，UI 绝对优先显示数字，不显示 Note。

---

## 2. 我认为提示功能更属于 `Sudoku` 还是 `Game`？为什么？

我采用了“**`Sudoku` 负责规则计算，`Game` 负责会话编排**”的协作方案。

- `Sudoku` 负责“纯领域规则”：
  - 候选集合计算；
  - 下一步可推定计算；
  - 冲突检测等。
- `Game` 负责“会话语义与状态路由”：
  - 当前是否探索中；
  - 提示应用到哪个局面（主局面 / 探索局面）；
  - 与历史、Undo/Redo、提示次数扣减策略协同。

这样能避免 UI 直接触碰规则细节，也能保证探索模式下提示仍然自洽。

---

## 3. 我如何实现探索模式？

我选择“**子会话探索 + 深拷贝隔离 + 双线性 history**”。

### 3.1 探索生命周期

在 `Game` 中新增：

- `startExplore()`：进入探索，创建子会话；
- `rollbackExplore()`：回到探索起点；
- `commitExplore()`：提交探索结果；
- `abortExplore()`：放弃探索结果；
- `isExploring()` / `getExploreStatus()`：读取探索状态。

### 3.2 冲突与失败记忆

- `Sudoku` 新增：
  - `getSignature()`：局面签名；
  - `hasConflict()`：是否冲突。
- 探索会话内维护 `failedSignatures`：
  - 出现冲突时记录当前签名为失败路径；
  - 当再次进入同签名局面时标记 `knownFailed`，并反馈用户。

---

## 4. 主局面与探索局面的关系是什么？

是“**复制对象（深拷贝）**”，不是共享对象。

- 进入探索时：
  - 主局面 `cur_sudoku` 深拷贝为 `startSudoku`；
  - 再创建 `sessionGame` 作为探索子会话。
- 探索期间：
  - 所有输入、撤销、重做都作用于 `sessionGame`。
- 提交时：
  - 把探索终局 `afterGrid` 合并回主局面。
- 放弃时：
  - 直接丢弃探索上下文，主局面不受影响。

此方案避免了引用污染，回滚逻辑也更稳定。

---

## 5. 我的 history 结构在本次作业中是否发生了变化？

发生了变化，从单线性扩展为“**双线性（主线 + 探索线）**”：

- 主会话继续使用 `undoStack / redoStack`。
- 探索会话拥有独立 `undoStack / redoStack`。
- 提交探索时：
  - 不回放探索全部步骤；
  - 压缩为主历史的一条 `EXPLORE_COMMIT` 记录（含 `beforeGrid/afterGrid`）。
- 主线 Undo/Redo 可以整体回退/重做这次探索提交。

我没有引入 DAG/树状合并，符合本次作业“不要求 DAG 合并”的范围。

## 6. Homework 1 中哪些设计在 Homework 2 暴露了局限？

### 6.1 UI 与领域边界不足

Homework 1 中一些行为更偏 UI 驱动（例如候选笔记、提示触发语义），到 Homework 2 加入 Hint/Explore 后，容易出现“UI 覆盖领域语义”的问题。

### 6.2 history 仅按单一会话设计

Homework 1 的 Undo/Redo 默认只有一条时间线。引入探索分支后，如果继续共用一个栈，会出现语义混乱（主流程与探索流程互相污染）。

### 6.3 局面身份缺失

Homework 1 没有“局面签名”概念，难以支持“失败路径记忆”。Homework 2 通过签名补上了这一点。

## 7. 如果重做一次 Homework 1，我会如何修改原设计？

如果从头重做，我会提前做三件事：

1. **把 `Sudoku` 做成更纯的规则核心**
   - 早期就固定“规则计算接口”（候选、冲突、完成态、签名）。

2. **把 `Game` 做成显式会话状态机**
   - 提前抽象“当前活动局面（active session）”概念，减少后续加探索时的侵入式改动。

3. **为历史记录预留动作类型扩展点**
   - 从一开始就让历史项支持 `type`（普通落子 / 批量提交 / 分支提交），便于后续演进。

- 探索未提交时不触发主局面胜利判定。
